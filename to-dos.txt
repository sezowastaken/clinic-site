Aşama 1 — Güvenli randevu sorgulama altyapısı

Öncelikle randevu modeline güvenli bir public referans eklenmeli:

Tahmin edilmesi zor benzersiz public_reference

Referansın public başvuru sonrasında kullanıcıya gösterilmesi
POST /api/public/appointment-lookup
Randevu kodu + telefon ile sorgulama
Rate limit
Minimal response
/randevu-sorgula public sayfası

İsim sadece sonucun içinde maskeli biçimde kullanılabilir:

Zehra H.

ama sorgu anahtarı olmamalı.

Aşama 2 — SMS bildirim sistemi

Randevu pending → confirmed olduğunda SMS gönderilmeli.

Burada yalnızca API çağırmak yeterli değil. Şunlar olmalı:

Tek bir SMS sağlayıcı adapter’ı
Sağlayıcı bilgileri .env içinde
Telefon numarası normalizasyonu
SMS gönderim kayıt tablosu
Gönderildi / başarısız / tekrar denendi durumu
Aynı randevu için tekrar tekrar onay SMS’i gönderilmemesi
SMS başarısız olsa bile randevunun onaylanmış kalması
Admin tarafında “SMS gönderilemedi, tekrar dene” seçeneği

SMS metni mümkün olduğunca genel olmalı:

Randevunuz onaylandı.
Tarih: ...
Saat: ...
Randevu kodu: ...

Prosedür adı gibi gereksiz hassas bilgiler SMS’e yazılmamalı. Bildirim SMS’i pazarlama mesajıyla veya reklam izniyle birleştirilmemeli. KVKK’nın 2025 tarihli kararında hizmet işlemleriyle ticari ileti rızalarının tek bir işlem içinde birleştirilmemesi gerektiği özellikle vurgulanıyor.

Aşama 3 — Müsaitlik backend’ine toplu işlem desteği

Arayüzden önce backend hazırlanmalı. Mevcut tek tek POST ve DELETE istekleri çoklu gün seçiminde onlarca request oluşturur.

Tek transaction içinde çalışan bir bulk endpoint eklenmeli:

PUT /api/admin/availability/bulk

Desteklemesi gereken işlemler:

Seçili günleri tamamen kapat
Seçili günleri tam gün müsait yap
Seçili günlere özel saat aralıkları uygula
Günlerin mevcut aralıklarını değiştir veya tamamen yenile

“Tam gün”ün ne olduğu net tanımlanmalı. Örneğin .env veya klinik ayarında:

CLINIC_DAY_START=09:00
CLINIC_DAY_END=18:00

Toplu işlem transaction içinde olmalı; günlerin yarısının güncellenip yarısının başarısız kalmaması gerekir.

Aşama 4 — Müsaitlik ekranının yeniden tasarlanması

Takvimde küçük noktalar yerine günün durumu doğrudan okunmalı:

Müsait değil
09:00–13:00
09:00–12:00, 14:00–17:00
Tam gün

Yeni etkileşim:

Günlere tıklayarak çoklu seçim
Seçili günlerin belirgin çerçeve/background ile gösterilmesi
Altta veya üstte toplu işlem çubuğu
Hızlı aksiyonlar:
Tam gün müsait
Müsait değil
Özel saat belirle
Tek gün seçildiğinde basit modal/drawer
Mobilde sağ kolon yerine tam ekran drawer

Mevcut “sebep” alanı gerçekten kullanılmayacaksa kaldırılmalı. Kullanılacaksa ayrı bir gün durumu tablosuyla kalıcı hâle getirilmeli; şu anki gibi görünüşte var fakat kaydedilmeyen bir alan bırakılmamalı.

Aşama 5 — Ortak tasarım sistemi

Public ve admin sayfaları tek promptta baştan yapılmamalı. Önce temel tasarım sistemi oluşturulmalı:

Renk ve tipografi token’ları
Button
Input / Select / Textarea
Card
Dialog / Drawer
Tooltip
StatusBadge
PageHeader
EmptyState
ErrorState
Skeleton
Toast

Yeni kütüphane kullanımı sınırlı tutulmalı:

Radix Primitives: erişilebilir dialog, popover ve tooltip gibi etkileşimler
TanStack Query: mevcut tekrarlanan useEffect + loading + error + refresh kodlarını merkezi API state yapısına taşımak
Lucide React: tutarlı ikonlar
Motion: yalnızca public tarafta ölçülü mikro animasyonlar

Radix erişilebilir, özelleştirilebilir düşük seviyeli bileşenlere odaklanıyor; TanStack Query ise React tarafında sunucu verisini çekme, cache ve invalidation yapısını merkezileştiriyor. Motion üretim amaçlı React animasyonları sunuyor.

Tam bir ağır UI framework’ü eklemek gerekli değil. Mevcut Tailwind tabanı korunabilir.
